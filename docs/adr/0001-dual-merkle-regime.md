# ADR-0001: Dual-Merkle Hashing Regime

**Status:** Accepted, 2026-05-21.
**Supersedes:** the pre-2026-05-21 single-tree implementation.

## Context

The contract `LandRegistryAnchor.verifyProof` uses sorted-pair Keccak-256
(matching the OpenZeppelin `MerkleProof` library and the EVM-native hash).
Cost: each proof verification is ~6,000 gas on mainnet — cheap.

The audit ledger `audit_events.payload_hash` is SHA-256 hex; the ledger has
been SHA-256 since the prototype's sibling-project provenance. Migrating
the chain itself to keccak would invalidate every previously-verified row
and force every auditor to import an EVM hash library — a high-friction
ask for civil-society reviewers.

The original implementation hand-rolled two incompatible trees: the
Python `compute_merkle_root` was index-ordered SHA-256, while the contract
expected sorted-pair keccak. Proofs generated by Python would fail
verification on-chain.

## Decision

Maintain **two parallel Merkle implementations** in the same module:

1. `compute_merkle_root` / `merkle_proof` (index-ordered SHA-256) — the
   off-chain integrity tree, used by chain verifiers, third-party auditors,
   and any Python/TS code that wants to recompute roots without a keccak
   library.

2. `compute_merkle_root_evm` / `merkle_proof_evm` (sorted-pair Keccak-256
   over `keccak(sha256_hex_leaf)` leaves) — the on-chain anchor tree,
   identical algorithm to the contract's `verifyProof`.

The anchor service commits the **EVM root** on-chain, but also records the
SHA-256 root in the `ANCHOR_COMMITTED` audit event so chain verifiers can
correlate. The public verifier (and printed QR codes) return the EVM-form
proof exclusively; an evaluator who wants to recompute it offline uses
`verify_merkle_proof_evm` in Python or `verifyMerkleProofEvm` in TypeScript,
both of which import only widely-available SHA-3 keccak.

## Consequences

**Positive:**

- The contract is the source of truth for verification; no implicit hash
  conversions inside Python or TypeScript code paths.
- A printed title QR contains keccak hex; offline verification works with
  any keccak library.
- The audit chain integrity tree stays SHA-256, requiring no migration of
  existing rows.

**Negative:**

- Two Merkle paths to maintain. Mitigated by `test_merkle_cross.py` which
  cross-checks both regimes against a published fixture.
- An auditor must understand the bridge (`keccak(sha256_hex_leaf)`).
  Documented prominently at the top of `app/audit/merkle.py`.

## Alternatives considered

- **Migrate the chain to keccak.** Rejected: invalidates the entire
  pre-existing ledger; forces non-EVM auditors to import an extra hash
  library; cosmetic gain only.
- **Commit BOTH roots on-chain.** Considered. Doubles tx gas cost for
  marginal benefit; we keep the SHA-256 root in the audit event payload
  instead, where it remains tamper-evident under the chain hash.
- **Use only the on-chain root, no off-chain Merkle.** Rejected: the chain
  verifier needs an integrity proof that doesn't depend on an EVM client.
