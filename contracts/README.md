# LandGuard Smart Contracts

Foundry · Solidity 0.8.24 · OpenZeppelin Contracts ^5.

Two on-chain contracts back the LandGuard public verifier:

| Contract | Purpose |
| --- | --- |
| `LandRegistryAnchor.sol` | Per-district Merkle anchors and proof verification |
| `MultiSigRegistrar.sol` | 3-of-5 named-signer custody of the registrar role |

## 4. LandRegistryAnchor

### 4.1 Storage

```solidity
mapping(uint16 districtId => mapping(bytes32 batchId => bytes32)) public anchors;
```

A single mapping keyed by `(districtId, batchId)`. Storing only the
Merkle root keeps gas cost flat regardless of how many events the batch
contains.

### 4.2 Interface

```solidity
function commitBatch(
    uint16 districtId,
    bytes32 batchId,
    bytes32 merkleRoot
) external onlyRole(REGISTRAR_ROLE) whenNotPaused;

function verifyProof(
    uint16 districtId,
    bytes32 batchId,
    bytes32 leaf,
    bytes32[] calldata proof
) external view returns (bool);

function pause() external onlyRole(DEFAULT_ADMIN_ROLE);
function unpause() external onlyRole(DEFAULT_ADMIN_ROLE);
```

### 4.3 Merkle conventions

- **Sorted-pair Keccak.** Every internal node is computed as
  `keccak256(min(a,b), max(a,b))`. This matches OpenZeppelin's
  `MerkleProof.verify` semantics, which means proofs can be verified
  off-chain with any standard library.
- **Leaves are `keccak256(sha256_hex_leaf)`** — see ADR-0001 for the
  rationale behind the dual hash function (off-chain SHA-256 for the
  ledger, on-chain Keccak for gas efficiency).

### 4.4 Events

```solidity
event AnchorCommitted(
    uint16 indexed districtId,
    bytes32 indexed batchId,
    bytes32 merkleRoot,
    address indexed registrar
);
```

The chain's event log is itself an audit trail — indexed on `districtId`
and `batchId` so an auditor can stream all batches for a district
without scanning the full chain.

### 4.5 Tests

`test/LandRegistryAnchor.t.sol` covers:

- Happy-path commit and re-verify
- Duplicate-batch reverts (`batchId` collision)
- Non-registrar caller reverts (`AccessControlUnauthorizedAccount`)
- Pause / unpause behaviour
- `verifyProof` against the same hash vectors the Python ledger uses
  (cross-language fixture in `test/fixtures/merkle_vectors.json`)

The cross-language test asserts that a proof generated by
`backend/app/audit/merkle.py:compute_merkle_root_evm()` verifies in
Solidity byte-for-byte. This is the load-bearing audit guarantee.

## 5. MultiSigRegistrar

### 5.1 Custody model

Five named signers:

| Signer | Role-name in `docs/CUSTODY.md` |
| --- | --- |
| 1 | MoLHUD (Ministry of Lands, Housing & Urban Development) |
| 2 | NITA-U (National IT Authority Uganda) |
| 3 | District Land Board (rotating per active pilot district) |
| 4 | LandGuard backend (auto-co-signer via `co_sign_daemon.py`) |
| 5 | Independent Observer (Makerere CSL recommended) |

A `commitBatch` proposal requires **3 of 5** confirmations to execute.
The auto-co-signer (signer 4) confirms automatically once the proposal
has passed at least one human confirmation — preventing it from acting
unilaterally.

### 5.2 Interface

```solidity
function proposeBatch(
    uint16 districtId,
    bytes32 batchId,
    bytes32 merkleRoot
) external onlyRole(PROPOSER_ROLE) returns (uint256 proposalId);

function confirmProposal(uint256 proposalId) external onlyRole(SIGNER_ROLE);

function executeProposal(uint256 proposalId) external;     // anyone

function revokeConfirmation(uint256 proposalId) external onlyRole(SIGNER_ROLE);
```

### 5.3 Events

```solidity
event ProposalCreated(uint256 indexed proposalId, ...);
event ProposalConfirmed(uint256 indexed proposalId, address signer);
event ProposalExecuted(uint256 indexed proposalId);
event ProposalRevoked(uint256 indexed proposalId, address signer);
```

The chain log surfaces the full custody trail — three distinct
transactions before a single `AnchorCommitted` lands on
`LandRegistryAnchor`. This is the showcase moment for Act 2.

### 5.4 Tests

`test/MultiSigRegistrar.t.sol` covers:

- Propose → confirm × 3 → execute (happy path)
- Below-threshold execute reverts
- Double-confirmation reverts
- Revoke-then-confirm flow
- Proposal expiry (7-day default) reverts on execute

## 6. Deployment

### 6.1 Local (Anvil)

```bash
anvil --silent &
forge script script/Deploy.s.sol \
  --rpc-url http://localhost:8545 \
  --private-key 0xac0974be... \
  --broadcast
```

The script writes `backend/app/blockchain/abi.json` (compiled ABI) and
`backend/data_store/contract_address.json` (deployed address) so the
backend can pick it up on startup.

For the multi-sig pilot:

```bash
MULTISIG_SIGNERS="0x...,0x...,0x...,0x...,0x..." \
MULTISIG_THRESHOLD=3 \
forge script script/DeployMultiSig.s.sol \
  --rpc-url http://localhost:8545 \
  --private-key 0xac0974be... \
  --broadcast
```

This deploys both contracts and transfers `REGISTRAR_ROLE` on
`LandRegistryAnchor` to the `MultiSigRegistrar` instance.

### 6.2 Sepolia

```bash
export SEPOLIA_RPC_URL=https://sepolia.infura.io/v3/...
export SEPOLIA_PRIVATE_KEY=0x...   # from KMS, never inline

forge script script/Deploy.s.sol \
  --rpc-url $SEPOLIA_RPC_URL \
  --private-key $SEPOLIA_PRIVATE_KEY \
  --broadcast \
  --verify \
  --etherscan-api-key $ETHERSCAN_API_KEY
```

Verification on Etherscan is required so evaluators can read the source
of the contract holding their land records.

### 6.3 Production (future EAC or MoICT&NG permissioned chain)

The contracts are EVM-only; no chain-specific code. To migrate:

1. Deploy `LandRegistryAnchor` and `MultiSigRegistrar` to the target
   chain via `forge script script/Deploy.s.sol`.
2. Update `backend/app/blockchain/anvil_client.py` → new
   `production_client.py` with the target RPC URL (or use
   `sepolia_client.py` as a template).
3. Set `BLOCKCHAIN_PROVIDER=production` and `RPC_URL=<chain rpc>` in
   backend env.
4. Verify the cross-language Merkle vectors pass on the new chain
   (`forge test --rpc-url <chain rpc>`).

## 7. Security posture

- **Solidity 0.8.24**: arithmetic overflow protection by default.
- **OpenZeppelin AccessControl**: role-based gating on `commitBatch`,
  `pause`, and threshold confirmation.
- **OpenZeppelin Pausable**: kill-switch for the registrar's anchor
  permission. Pausing does NOT invalidate already-anchored batches —
  the public verifier continues to work in read-only mode.
- **No upgradeability**: contracts are intentionally not proxied. A
  contract migration requires re-deploying and re-anchoring — this
  forces a transparent, audited rotation rather than a silent upgrade.
- **No ERC-20 / ERC-721 dependencies**: land titles are not tokens.
  Tokenisation would conflict with Ugandan land-titling law (a title is
  a registered document, not a transferable asset).
- **External audits**: scheduled post-Mityana-pilot. Budget allocated
  (~ UGX 5M) for independent third-party review of both contracts plus
  a fuzzing pass via Echidna or Foundry's invariant testing.

## 9. Common operations

```bash
# Add a new signer to the multi-sig
# (requires existing 3-of-5 vote on grantRole(SIGNER_ROLE, new_signer))
cast send $MULTISIG \
  "grantRole(bytes32,address)" \
  $(cast keccak "SIGNER_ROLE") \
  $NEW_SIGNER \
  --rpc-url $RPC_URL --private-key $PROPOSER_KEY

# Pause new anchoring (admin-only emergency)
cast send $ANCHOR "pause()" \
  --rpc-url $RPC_URL --private-key $ADMIN_KEY

# Replay a batch off-chain and verify against the chain
forge test --match-test "test_verifyProof_against_python_vector" -vvv

# Generate a fresh Merkle proof from Python and verify it via forge
python ../backend/scripts/generate_merkle_proof.py \
  --district 3 --title "UG-MIT-T00007/2026" > /tmp/proof.json
forge test --match-test test_verifyProof_external -vv
```
